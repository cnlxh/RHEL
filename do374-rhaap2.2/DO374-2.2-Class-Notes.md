# 第一章： 使用RHAAP2开发Playbook

## RHAAP 介绍

**红帽 Ansible ⾃动化平台 2 组件：**

1. **Ansible Core**

红 帽 Ansible ⾃动化平台 2.2 提供了 Ansible Core 2.13

Ansible Core 提供⽤于运⾏ Ansible Playbook 的基本功能。⾃动化代码所需的关键功能，如循环、条件和其他Ansible 命令。 还提供了驱动⾃动化所需的框架和基本命令⾏⼯具。

2. **Ansible 内容集合**

Ansible Core 本⾝仅限于由 `ansible.builtin` 这个Ansible 内容集合提供的⼀⼩组模块，和其他的模块分开发展，⼤多数模块重新整理为单独的 Ansible 内容集，⽤⼾可以根据⾃⾝需求灵活地选择不同版本的集合或不同系列的集合。 它也让开发⼈员能够独⽴于 Ansible 本⾝的节奏更新他们的模块。

3. **⾃动化内容导航器**

rhaap2 提供了一个名为`ansible-navigator` 的命令来开发测试playbook，这个命令替代了以前的`ansible-playbook`、`ansible-inventory`、`ansible-config`、`ansible-doc`等

`navigator` 通过在容器中运⾏ playbook，将运⾏ Ansible 的`控制节点`与运⾏它的`⾃动化执⾏环境`分隔开来，这样就可以用不同的`⾃动化执⾏环境`来应对不同的python环境需求

4. **⾃动化执⾏环境**

⾃动化执⾏环境是⼀种容器镜像，其包含 Ansible Core、Ansible 内容集，以及运⾏ playbook 所需的任何 Python 库、可执⾏⽂件或其他依赖项。

使⽤ ansible-navigator 运⾏ playbook 时，可以选择不同的`⾃动化执⾏环境`来为不同的`python`环境提供支持，如果需要，也可以用`ansible-builder` 来⾃⾏创建⾃定义执⾏环境。

5. **⾃动化控制器**

⾃动化控制器以前称为红帽 Ansible Tower，提供 Web UI 和 REST API ⽤于配置、运⾏和评估⾃动化作业

6. **⾃动化中⼼**

⾃动化中⼼提供⼀种管理和分发⾃动化内容的⽅式，类似于一个在线的仓库，可以从仓库下载内容，可通过 `console.redhat.com` 上的公共服务访问红 帽 Ansible 认证内容集合，或者部署私有 `Automation Hub`

## 使用ansible-navigator运行Playbook

### ansible-navigator

RHAAP2中提供了`ansible-navigator` 命令来运行Playbook，不再使用`ansible-playbook`等命令

以前执行Playbook：

```bash
[root@lixiaohui ~]# ansible-playbook -i inventory playbook.yml 
```

现在执行Playbook：

```bash
[root@lixiaohui ~]# ansible-navigator run playbook.yml -i inventory -m stdout
```

需要注意这个`-m stdout`,如果不加这个-m参数，会进入交互式运行的状态，而不是默认输出到屏幕的方式

### 使用自动化执行环境来提⾼可移植性

⾃动化执⾏环境是⼀种容器镜像，借助⾃动化执⾏环境，可以避免在⾃动化控制器上创建多个 Python 虚拟环境。rhaap2使⽤多个⾃动化执⾏环境，⽽不是多个 Python 虚拟环境

如果使⽤ ansible-navigator 运⾏ playbook，并且不指定特定的⾃动化执⾏环境，ansible-navigator 会⾃动尝试从 `registry.redhat.io` 调取默认的⾃动化执⾏环境，对于rhaap2来说，这个环境是：`registry.redhat.io/ansible-automation-platform-22/ee-supported-rhel8:latest`

在运行Playbook时，可以用`--eei`来选择特定的容器镜像以⽤作⾃动化执⾏环境，也可以用`-pp`选项来控制 ansible-navigator 如何调取容器镜像。 当此选项的值设置为 `missing`时，⾃动化内容导航器仅在本地系统上不存在镜像时才拉取容器镜像。

### 安装ansible-navigator

Ansible 是非侵入式的自动化平台，所以只需要在控制节点安装即可，需要注意的是，安装需要开通相应的红帽订阅

```bash
[root@lixiaohui ~]# yum install ansible-navigator -y
```

### 配置受管主机的⾝份验证

ansible-navigator需要能够登录受管主机并获得这些主机的超级⽤⼾特权。 实施此操作的最简单⽅法是对帐⼾使⽤基于 SSH 密钥的⾝份验证，该帐⼾应当允许通过 sudo 进⾏特权升级，⽽⽆需输⼊密码。

首先需要在控制节点上给自动化用户生成ssh密钥，并分发给远程的用户

```bash
[user1@lixiaohui ~]$ ssh-keygen
[user1@lixiaohui ~]$ ssh-copy-id user1@remotehost
```

然后在控制节点配置ansible.cfg

```bash
[user1@lixiaohui ~]$ vim /etc/ansible/ansible.cfg
```

```ini
[defaults]
remote_user=user1
```

需要在所有的受管主机上配置sudo提权

```bash
[root@lixiaohui ~]# vim /etc/sudoers
...
user1 ALL=ALL(ALL) NOPASSWD: ALL
```

将 ansible-navigator 和 ansible-playbook 与受管主机之间进⾏⾝份验证的主要区别在于，ansible-navigator 在容器内运⾏ playbook，⽆法访问 ~/.ssh ⽬录。 但是，如果在控制节点上运⾏ ssh-agent 来存储 SSH 私钥，则在运⾏ ansible-navigator 时，它可以⾃动将这些密钥提供给执⾏环境。

```bash
[user1@lixiaohui ~]$ eval $(ssh-agent)
[user1@lixiaohui ~]$ ssh-add
```

#### Git pull和Git fetch

**相同点：**

都是用来从远程仓库获取更新的命令

**不同点**

`git fetch` 命令会从远程仓库拉取最新的提交，但不会自动合并到你当前的工作分支上，这个命令通常用于查看远程仓库的更新情况，以及在本地分析远程分支的变化，然后决定如何处理，你需要自己执行`git merge`操作来合并远程和本地

`git pull` 命令实际上是两个步骤的结合：首先执行 `git fetch`，然后自动执行 `git merge`，将远程分支的更新合并到你当前的工作分支上

以上的merge操作，如果有冲突，会提醒你，需要你处理然后才会merge

#### 创建Git 库

`git init` 用于在当前目录下创建一个新的 Git 仓库，Git 会在该目录下创建一个 .git 目录，该目录包含了 Git 仓库的所有信息，包括版本历史、配置和对象数据库等

`git init --bare` 用于创建一个裸仓库，也就是没有工作目录的仓库，裸仓库是不包含工作目录的 Git 仓库，只包含 Git 版本历史和对象数据库，通常用作远程仓库（例如在服务器上托管的仓库）

总的来说，`git init` 用于创建普通的 Git 仓库，而 `git init --bare` 用于创建裸仓库，通常用作远程仓库

### 使⽤ Git 分⽀和引⽤

当我们的1.0向线上发布时，我们就需要创建1.1.1的分支进行开发

```bash
[root@lixiaohui ~]# lxh-branch commit
```

**将lxh-branch分支合并到master主线**

git merge可以合并，但是如果遇到冲突，就需要人工干预，具体如下：

```bash
[root@lixiaohui ~]# git merge lxh-branch
Auto-merging LICENSE.txt
CONFLICT (content): Merge conflict in LICENSE.txt
Automatic merge failed; fix conflicts and then commit the result.

[root@lixiaohui ~]# cat LICENSE.txt
<<<<<<< HEAD
master line 1
=======
lxh-branch line 1
>>>>>>> lxh-branch
```

手工编辑文件，把哪里需要保留，哪里需要删除处理好，并把结果commit

```bash
[root@lixiaohui ~]# cat LICENSE.txt
master line 1
lxh-branch line 1
[root@lixiaohui ~]# git add LICENSE.txt
[root@lixiaohui ~]# git commit -m 'merge lxh-branch'
[master afba5d8] merge lxh-branch
```

### 在 Git 中构建 Ansible 项⽬

每个 Ansible 项⽬都应拥有⾃⼰的 Git 存储库，每个项目的文件结构应该遵守以下布局：

```text
production                # inventory file for production servers
staging                   # inventory file for staging environment

group_vars/
   group1.yml             # here we assign variables to particular groups
   group2.yml
host_vars/
   hostname1.yml          # here we assign variables to particular systems
   hostname2.yml

library/                  # if any custom modules, put them here (optional)
module_utils/             # if any custom module_utils to support modules, put them here (optional)
filter_plugins/           # if any custom filter plugins, put them here (optional)

site.yml                  # main playbook
webservers.yml            # playbook for webserver tier
dbservers.yml             # playbook for dbserver tier
tasks/                    # task files included from playbooks
    webservers-extra.yml  # <-- avoids confusing playbook with task files
```

#### ⻆⾊和 Ansible 内容集合

**Ansible 内容集合:** 通常存储在 collections ⽬录的⼦⽬录中

**角色:** 存储在 roles ⽬录的⼦⽬录中

两个⽬录中的⼀个或两个中可能还有⼀个 requirements.yml ⽂件

#### 配置 Git 以忽略⽂件

在 Ansible 项⽬的顶级⽬录中添加 `.gitignore` ⽂件，`.gitignore` ⽂件中的每⼀⾏代表⼀个要匹配的模式，它将决定 Git 是否忽略存储库中的某个⽂件。 通常，与⽂件名匹配的⾏表⽰ Git 忽略的⽂件。 以感叹号 (!) 开头的⾏表⽰ Git 不应忽略的⽂件，即使它们与先前的模式匹配。

.gitignore ⽂件示例:


## 实施推荐的 Ansible 做法

### 增加可读性

1. 添加注释

2. 逻辑之间采用空行隔开

3. 始终为play和task起一个有意义的名称

4. 使用统一风格，例如相同的空格数量，名称变量命名规范，注释要求等

5. 利用角色提高效率

6. 坚持使用YAML原生语法，不要把参数写在一行

推荐使用的原生语法如下：

```yaml
- name: Postfix is installed and updated
  ansible.builtin.yum:
    name: postfix
    state: latest
  notify: update_postfix
- name: Postfix is running
  ansible.builtin.service:
    name: postfix
    state: started
```

以下为不推荐的写法：

```yaml
- name: Postfix is installed and updated
  ansible.builtin.yum: name=postfix state=latest
  notify: restart_postfix
- name: Postfix is running
  ansible.builtin.service: name=postfix state=started
```

变量的名字应该能充分体现出其内容

提高变量的命名可读性非常重要，例如使用`apache_tls_port`作为变量名，而不是使用`p`作为变量

如果⻆⾊的名称是 `myapp`，则为变量添加前缀 `myapp_`，提高识别度


### 使用功能模块而不是命令

Ansible Playbook具有幂等效果，但是如果使用a`nsible.builtin.command`、`ansible.builtin.shell`、`ansible.builtin.raw`，就会破坏幂等效果，应减少使用这3个特殊的模块

在调试时，可以时常使用`ansible.builtin.debug` 这个模块

大多数模块都有一个`state`之类的参数值，我们应该`显式`的使用state，而不是使用默认行为，这有助于提高阅读，也可以避免后期默认行为变更带来负面影响

### 动态清单介绍

在被管理主机不是一尘不变时，可以用动态清单来获取具体的主机列表，然后再运行playbook

也可以用`group_by`模块来动态生成主机组，提高playbook运行效率

```yaml
- name: Generate dynamic groups
  hosts: all
  tasks:
    - name: Generate dynamic groups based on architecture
      ansible.builtin.group_by:
        key: arch_"{{ ansible_facts['architecture'] }}"
- name: Configure x86_64 systems
  hosts: arch_x86_64
  tasks:
    - name: First task for x86_64 configuration
 ```

### 测试任务的结果

除了使用`debug`模块之外，还可以使用`uri`模块用于验证`web类型`的服务是否正常

```yaml
- name: Start web server
  ansible.builtin.service:
    name: httpd
    status: started
- name: Check web site from web server
  ansible.builtin.uri:
    url: http://{{ ansible_fqdn }}
    return_content: true
  register: example_webpage
  failed_when: example_webpage.status != 200
```

### Block 和 Rescue

```yaml
- block:
    - name: Check web site from web server
      ansible.builtin.uri:
        url: http://{{ ansible_fqdn }}
        return_content: true
      register: example_webpage
      failed_when: example_webpage.status != 200
  rescue:
    - name: Restart web server
      ansible.builtin.service:
        name: httpd
        status: restarted
```

定期针对 Ansible Core 的最新版本测试playbook。 此测试有助于避免在 Ansible 模块和功能不断演变时遭遇问题
