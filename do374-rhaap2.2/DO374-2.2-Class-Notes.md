# 第一章： 使用RHAAP2开发Playbook

## RHAAP 介绍

**红帽 Ansible ⾃动化平台 2 组件：**

1. **Ansible Core**

红 帽 Ansible ⾃动化平台 2.2 提供了 Ansible Core 2.13

Ansible Core 提供⽤于运⾏ Ansible Playbook 的基本功能。⾃动化代码所需的关键功能，如循环、条件和其他Ansible 命令。 还提供了驱动⾃动化所需的框架和基本命令⾏⼯具。

2. **Ansible 内容集合**

Ansible Core 本⾝仅限于由 `ansible.builtin` 这个Ansible 内容集合提供的⼀⼩组模块，和其他的模块分开发展，⼤多数模块重新整理为单独的 Ansible 内容集，⽤⼾可以根据⾃⾝需求灵活地选择不同版本的集合或不同系列的集合。 它也让开发⼈员能够独⽴于 Ansible 本⾝的节奏更新他们的模块。

3. **⾃动化内容导航器**

rhapp2 提供了一个名为`ansible-navigator` 的命令来开发测试playbook，这个命令替代了以前的`ansible-playbook`、`ansible-inventory`、`ansible-config`、`ansible-doc`等

`navigator` 通过在容器中运⾏ playbook，将运⾏ Ansible 的`控制节点`与运⾏它的`⾃动化执⾏环境`分隔开来，这样就可以用不同的`⾃动化执⾏环境`来应对不同的python环境需求

4. **⾃动化执⾏环境**

⾃动化执⾏环境是⼀种容器镜像，其包含 Ansible Core、Ansible 内容集，以及运⾏ playbook 所需的任何 Python 库、可执⾏⽂件或其他依赖项。

使⽤ ansible-navigator 运⾏ playbook 时，可以选择不同的`⾃动化执⾏环境`来为不同的`python`环境提供支持，如果需要，也可以用`ansible-builder` 来⾃⾏创建⾃定义执⾏环境。

5. **⾃动化控制器**

⾃动化控制器以前称为红帽 Ansible Tower，提供 Web UI 和 REST API ⽤于配置、运⾏和评估⾃动化作业

6. **⾃动化中⼼**

⾃动化中⼼提供⼀种管理和分发⾃动化内容的⽅式，类似于一个在线的仓库，可以从仓库下载内容，可通过 `console.redhat.com` 上的公共服务访问红 帽 Ansible 认证内容集合，或者部署私有 `Automation Hub`

## 使用ansible-navigator运行Playbook

### ansible-navigator

RHAAP2中提供了`ansible-navigator` 命令来运行Playbook，不再使用`ansible-playbook`等命令

以前执行Playbook：

```bash
[root@lixiaohui ~]# ansible-playbook -i inventory playbook.yml 
```

现在执行Playbook：

```bash
[root@lixiaohui ~]# ansible-navigator run playbook.yml -i inventory -m stdout
```

需要注意这个`-m stdout`,如果不加这个-m参数，会进入交互式运行的状态，而不是默认输出到屏幕的方式

### 使用自动化执行环境来提⾼可移植性

⾃动化执⾏环境是⼀种容器镜像，借助⾃动化执⾏环境，可以避免在⾃动化控制器上创建多个 Python 虚拟环境。rhapp2使⽤多个⾃动化执⾏环境，⽽不是多个 Python 虚拟环境

如果使⽤ ansible-navigator 运⾏ playbook，并且不指定特定的⾃动化执⾏环境，ansible-navigator 会⾃动尝试从 `registry.redhat.io` 调取默认的⾃动化执⾏环境，对于rhapp2来说，这个环境是：`registry.redhat.io/ansible-automation-platform-22/ee-supported-rhel8:latest`

在运行Playbook时，可以用`--eei`来选择特定的容器镜像以⽤作⾃动化执⾏环境，也可以用`-pp`选项来控制 ansible-navigator 如何调取容器镜像。 当此选项的值设置为 `missing`时，⾃动化内容导航器仅在本地系统上不存在镜像时才拉取容器镜像。

### 安装ansible-navigator

Ansible 是非侵入式的自动化平台，所以只需要在控制节点安装即可，需要注意的是，安装需要开通相应的红帽订阅

```bash
[root@lixiaohui ~]# yum install ansible-navigator -y
```

### 配置受管主机的⾝份验证

ansible-navigator需要能够登录受管主机并获得这些主机的超级⽤⼾特权。 实施此操作的最简单⽅法是对帐⼾使⽤基于 SSH 密钥的⾝份验证，该帐⼾应当允许通过 sudo 进⾏特权升级，⽽⽆需输⼊密码。

首先需要在控制节点上给自动化用户生成ssh密钥，并分发给远程的用户

```bash
[user1@lixiaohui ~]$ ssh-keygen
[user1@lixiaohui ~]$ ssh-copy-id user1@remotehost
```

然后在控制节点配置ansible.cfg

```bash
[user1@lixiaohui ~]$ vim /etc/ansible/ansible.cfg
```

```ini
[defaults]
remote_user=user1
```

需要在所有的受管主机上配置sudo提权

```bash
[root@lixiaohui ~]# vim /etc/sudoers
...
user1 ALL=ALL(ALL) NOPASSWD: ALL
```

将 ansible-navigator 和 ansible-playbook 与受管主机之间进⾏⾝份验证的主要区别在于，ansible-navigator 在容器内运⾏ playbook，⽆法访问 ~/.ssh ⽬录。 但是，如果在控制节点上运⾏ ssh-agent 来存储 SSH 私钥，则在运⾏ ansible-navigator 时，它可以⾃动将这些密钥提供给执⾏环境。

```bash
[user1@lixiaohui ~]$ eval $(ssh-agent)
[user1@lixiaohui ~]$ ssh-add
```

### 运行ansible-navigator

|早期命令|现在命令|
|-|-|
|ansible-config|ansible-navigator config|
|ansible-doc|ansible-navigator doc|
|ansible-inventory|ansible-navigator inventory|
|ansible-playbook|ansible-navigator run|

**ansible-navigator的子命令：**

在交互式会话中运⾏⼦命令时，键⼊ : 可启动命令，如 :config

|子命令|描述|
|-|-|
|collections|获取有关已安装集合的信息|
|config|检查当前的 Ansible 配置|
|doc|检查插件的 Ansible ⽂档|
|help|显⽰ ansible-navigator 的详细帮助|
|images|检查执⾏环境|
|inventory|浏览清单|
|log|查看当前的⽇志⽂件|
|open|在⽂本编辑器中打开当前⻚⾯|
|replay|重播 playbook 构件|
|run|运⾏ playbook|

**检查以前的 Playbook 运⾏**

⾃动化内容导航器提供了⼀项新的重播功能，可显⽰上⼀次 playbook 运⾏的输出，Playbook默认会生成构件，也就是`Playbookname-artifact-xxxxx.json`的文件，从命令⾏使⽤ ansible-navigator replay 命令会读取构件以检查上⼀次 playbook 运⾏

重播⽂件的⼀个优点是，如果 playbook 运⾏失败，可以与其他开发⼈员或⽀持团队共享其重播⽂件。 他们可以继续进⾏故障排除，⽽⽆需访问项⽬⽬录中的所有⽂件，如 playbook、⻆⾊、清单⽂件和变量⽂件等。

## 使⽤ Git 管理 Ansible 项⽬资料

Git⼯作树中的⽂件可能处于三种状态：

|状态|描述|
|-|-|
|已修改|已编辑了⼯作树中⽂件的副本，它与存储库中的最新版本不同|
|已暂存|将修改后的⽂件添加到了已更改⽂件列表中，以作为⼀个集合⼀起提交，但还没有提交暂存的⽂件|
|已提交|已将修改后的⽂件提交到本地存储库|

将⽂件提交到本地存储库后，该⽂件可以推送到远程存储库或由远程存储库拉取。
